Chapter 01
🌐 Foundation

What is Active Directory?

The enterprise backbone that controls who you are, what you can access, and how your computer talks to the network.

📖
Definition
The official, beginner-friendly explanation

Active Directory (AD) is a directory service developed by Microsoft that runs on Windows Server. Think of it as a massive, intelligent phonebook for your entire company — it stores information about every user, computer, group, printer, and policy in the organization, and uses that information to control who can log in, what they can access, and what rules apply to their machine.

Directory Service Authentication Authorization LDAP-Based Windows Server
💡
Why Microsoft Created AD
The problem it solves
🔥

The Problem (Before AD)

Every computer had its own local user database. An IT admin managing 5,000 computers had to create the same user account 5,000 times. Password changes were a nightmare. No central control existed. Security was impossible to enforce consistently.

✅

The Solution (AD)

Microsoft released Active Directory with Windows 2000 Server. One central database. Create a user once, they can log into any computer. Enforce security policies across every machine from one place. Change a password once — it works everywhere.

🌍
Real-World Analogy
Makes the abstract concrete
🏢
Active Directory is like a Corporate Badge System

Imagine a large hospital. Every employee gets a badge. That badge has your name, role (doctor, nurse, admin), department, and access level. At every door — pharmacy, ICU, server room — the badge reader checks a central database: "Is this person allowed in here?" That central database is Active Directory. The badge is your domain user account. The door readers are the computers and applications checking your permissions.

Real Enterprise Example

Contoso Corporation has 12,000 employees across 8 countries. Their Active Directory setup:

Top-level container. Contains all domains. Security boundary. One schema shared across all domains.

👹 Attacker's Perspective
  • AD is the ultimate prize — compromise the DC = own the entire domain
  • AD stores password hashes in NTDS.dit — gold mine for credential attacks
  • Misconfigured AD permissions (ACLs) provide lateral movement paths
  • AD trusts between domains can allow cross-domain attacks
  • Kerberoasting, AS-REP Roasting, Pass-the-Hash all target AD
🛡️ Defender's Perspective
  • Harden Domain Controllers — limit who can access them physically and remotely
  • Enable audit logging on all DC events (logon, object changes)
  • Use tiered administration model (Tier 0/1/2)
  • Monitor for LDAP queries from non-admin accounts
  • Enable Protected Users security group for privileged accounts

Chapter 02
🖥️ Core Infrastructure

The Domain Controller

The brain of Active Directory. The most critical server in any Windows network — and the most targeted.

A Domain Controller (DC) is a Windows Server running the Active Directory Domain Services (AD DS) role. It is the central authority that authenticates users and computers, stores the AD database, enforces policies, and replicates data to other DCs. Without a DC, users cannot log in to the domain.

⚙️
Services Running on a DC

The KDC issues Ticket Granting Tickets (TGTs) and Service Tickets for Kerberos authentication. When you log in, your password hash is used to prove your identity to the KDC, which issues you a TGT — your "passport" for the session. Every subsequent service access uses this ticket rather than your password.

LDAP (port 389, LDAPS port 636) is the protocol used to query and modify the AD directory. Applications use LDAP to search for users, groups, and attributes. Attackers use LDAP enumeration to map the entire AD structure. Commands like ldapsearch and tools like BloodHound use LDAP extensively.

AD requires DNS to locate DCs and services. The DC runs an AD-integrated DNS zone, meaning DNS records are stored in AD and replicated automatically. Clients use DNS SRV records like _kerberos._tcp.domain.com to find the DC. DNS poisoning can redirect authentication traffic — a critical attack vector.

All GPO files are stored in the SYSVOL shared folder on the DC (\\DC\SYSVOL\domain\Policies\). When a machine boots or a user logs in, it contacts the DC, downloads applicable GPOs, and applies them. SYSVOL is replicated between DCs using DFSR. Attackers look for credentials and scripts stored in SYSVOL.

The heart of AD. Stored at C:\Windows\NTDS\NTDS.dit. Contains every object in the domain — all users, their password hashes (NT hashes, Kerberos keys), groups, computers, and more. Protected by the SYSTEM key. If an attacker dumps NTDS.dit + SYSTEM hive, they can crack every password in the domain offline.

🎤
Interview Questions
Expect these in SOC, Pentester, and AD Admin interviews
Q1
What happens if the PDC Emulator goes down?
Users may experience issues with password changes not propagating immediately. Kerberos authentication may fail if clocks drift beyond 5 minutes (since the PDC Emulator is the authoritative time source). GPO application may also be affected. Mitigation: seize the FSMO role to another DC using ntdsutil.
Q2
What's the difference between a Primary and Backup DC?
In modern AD (Windows 2000+), all DCs are multi-master — they can all handle writes and replicate to each other. The old Primary/Backup DC distinction was from Windows NT 4.0. Today, we have FSMO role holders for specific unique operations, but all DCs are peers for most functions.
Q3
Why is the Domain Controller the highest-priority target for attackers?
The DC holds NTDS.dit which contains all password hashes for every domain user. Compromising it gives: (1) every user's credentials, (2) ability to create golden/silver tickets, (3) complete control of domain policy, (4) ability to persist via DCSync or skeleton keys. Owning the DC = owning the entire domain.
👹 Attacker's Perspective
  • DCSync Attack: Mimic a DC replication request to extract all password hashes without touching NTDS.dit directly
  • NTDS.dit Dump: Volume Shadow Copy + secretsdump to extract offline
  • Skeleton Key: Patch LSASS on the DC so any password works for any user
  • Golden Ticket: Forge TGTs using the KRBTGT hash — persistent for 10 years
🛡️ Defender's Perspective
  • Restrict DC access to Tier 0 admins only — enforce jump servers
  • Enable Credential Guard to protect LSASS
  • Monitor for Directory Replication Service events (Event 4662)
  • Regularly rotate the KRBTGT account password (invalidates golden tickets)
  • Deploy Microsoft Defender for Identity (MDI) on DCs
Chapter 03
📦 Core Concepts

Active Directory Objects

Every entity in your environment — users, computers, groups, printers — is an AD object with attributes, a unique identifier (SID), and a distinguished name (DN).

👤

User Account

Represents a person or service. Has a SAM account name, UPN, password hash, SID. Can be member of groups. Assigned permissions via groups.

Security Principal
💻

Computer Account

Every domain-joined PC has a computer account ending in $ (e.g., WS01$). Has its own password, rotated every 30 days. Can authenticate to the domain like users.

Machine Identity
👥

Security Group

Containers for users/computers. Used to assign permissions to resources. Types: Domain Local, Global, Universal. Nest groups for scalability.

Access Control
🔧

Service Account

Special accounts for services/applications (IIS, SQL, Exchange). Often over-privileged. High-value targets for Kerberoasting. Use MSA/gMSA instead.

⚠ High Risk
📁

Organizational Unit

Container for organizing objects. GPOs are linked to OUs. Admins can delegate control at OU level. Not a security boundary — just organizational.

Container
🖨️

Printer / Shared Resource

Published printers and shared resources appear in AD, enabling users to find them. Less common in modern environments but still present in legacy setups.

Resource
🔍
Inside a User Object — Key Attributes
# PowerShell: Get all attributes of a user object
Get-ADUser -Identity "alice" -Properties * | Select-Object *

# Key attributes returned:
SamAccountName      : alice                   # Login name (pre-Win2000)
UserPrincipalName   : alice@corp.local         # Email-style login (UPN)
DistinguishedName   : CN=alice,OU=Eng,DC=corp,DC=local  # Full path in AD
ObjectSID           : S-1-5-21-2837952793-2699123456-1001  # Unique security ID
MemberOf            : {CN=Developers,OU=Groups...}  # Group memberships
PasswordLastSet     : 06/15/2025                # When password was changed
LastLogonDate       : 06/25/2025                # Last successful logon
BadPwdCount         : 0                         # Failed login attempts
ServicePrincipalName: {}                        # If set: Kerberoastable!
🔑

Understanding the Security Identifier (SID)

Every security principal (user, group, computer) has a SID: S-1-5-21-[Domain]-[RID]. The SID never changes even if you rename the user. Windows uses SIDs (not names) for all access control decisions. Well-known SIDs: S-1-5-18 = SYSTEM, S-1-5-21-...-500 = Built-in Administrator, S-1-5-21-...-512 = Domain Admins group. Attackers target accounts with high-value SIDs for privilege escalation.

🚩 HTB Checkpoint Enumerating AD Objects

Once you have initial access to a domain-joined machine, enumerate objects immediately:

▶Get-ADUser -Filter * -Properties ServicePrincipalName | Where {$_.ServicePrincipalName} — Find Kerberoastable accounts
▶Get-ADGroupMember "Domain Admins" — Who's a DA?
▶BloodHound + SharpHound: Graph all object relationships — finds attack paths automatically
▶Look for: stale accounts (LastLogon > 90 days), weak passwords, over-permissioned service accounts
Chapter 04
🔐 Security Core

Authentication — Kerberos & NTLM

How does AD know it's really you? Two protocols handle this: Kerberos (the modern, secure choice) and NTLM (the legacy fallback that attackers love to abuse).

Kerberos v5 is the primary authentication protocol in Active Directory. Named after the three-headed dog guarding the underworld, it involves three parties: the Client, the Key Distribution Center (KDC) on the DC, and the Service. No passwords travel the network — only encrypted tickets.

🎣

Kerberoasting

Any authenticated user can request a Service Ticket for any account with a ServicePrincipalName (SPN). The ST is encrypted with the service account's password hash. Attacker requests it and cracks offline. Mitigation: use long random passwords or gMSA for service accounts.

💀

Pass-the-Hash (NTLM)

NTLM only needs the NT hash — not the plaintext password. After dumping LSASS or NTDS.dit, an attacker uses the hash directly to authenticate as that user. Tools: pth-winexe, impacket, mimikatz. Mitigation: enable Protected Users group, disable NTLM where possible.

👹 Attacker's Perspective
  • Kerberoasting: Request service tickets for SPN accounts, crack offline
  • AS-REP Roasting: Target accounts with pre-auth disabled
  • Pass-the-Hash: Use NTLM hashes instead of passwords
  • Golden Ticket: Forge TGTs with stolen KRBTGT hash (persists 10 years)
  • Silver Ticket: Forge Service Tickets with service account hash
🛡️ Defender's Perspective
  • Require Kerberos pre-authentication for all accounts
  • Use Managed Service Accounts (gMSA) — auto-rotating 240-char passwords
  • Disable NTLM where not needed (use NTLMv2 minimum if needed)
  • Monitor Event ID 4769 (TGS requests) for unusual volume
  • Rotate KRBTGT password twice after any suspected compromise

From the moment you press Enter on your password to the desktop appearing — here's every step Windows takes, every process involved, and every security check performed.

Windows Domain Login — Internal Step-by-Step Process
⚠️
LSASS — The Gatekeeper (and Attacker Target)
💉

Why LSASS is the #1 Attack Target

  • LSASS stores NT hashes, Kerberos tickets, and cleartext credentials in memory
  • Mimikatz (sekurlsa::logonpasswords) dumps everything from LSASS memory
  • Windows Defender Credential Guard moves credentials into an isolated VM-based container — prevents LSASS dumps
  • Windows 11+ enables LSASS Protection (PPL) by default — requires kernel driver to bypass
  • Detection: Event ID 4656 / 10 (Sysmon) — handle on lsass.exe from unexpected process
🚩 HTB Checkpoint Credential Dumping Techniques

After gaining admin access on a domain-joined machine:

▶mimikatz # sekurlsa::logonpasswords — Dump LSASS credentials from memory
▶mimikatz # lsadump::dcsync /user:Administrator — DCSync to get DA hash without touching disk
▶impacket-secretsdump DOMAIN/admin@DC01 — Remote NTDS.dit dump
▶OPSEC tip: Use procdump -ma lsass.exe then offline mimikatz to reduce EDR detection

⚠️

Common Mistake: Same Local Admin Password

If all machines in the domain share the same local Administrator password (common in old environments), an attacker who dumps one SAM can use that hash to access every machine. Solution: LAPS (Local Administrator Password Solution) — rotates and randomizes local admin passwords per machine.

💡

The Built-in Administrator (RID 500)

Every Windows machine has a local Administrator account with RID 500. This account cannot be locked out. Even if renamed, tools identify it by its SID ending in -500. Always disable it or set a strong, unique password. It's often the target of local credential attacks.

Q1
Can a domain user log in if the DC is unavailable?
Yes — if the user has previously logged into that machine, Windows caches their credentials locally (up to 10 cached logons by default, controlled by the Interactive logon: Number of previous logons to cache policy). The cached verifier (DCC2 hash) allows offline login. Attackers target these cached hashes with tools like cachedump.
Q2
What is LAPS and why is it important?
Local Administrator Password Solution (LAPS) automatically sets a unique, random password for the local Administrator account on each domain-joined machine and stores it in AD (readable only by authorized admins). This prevents lateral movement via shared local admin credentials — one of the most common attack paths in enterprise environments.
Chapter 07
📋 Policy Control

Group Policy Objects (GPO)

The mechanism by which administrators enforce security settings, software configurations, and restrictions across every user and computer in the domain — from one central location.

A Group Policy Object (GPO) is a collection of settings that administrators apply to users or computers in an AD domain. GPOs are linked to Sites, Domains, or OUs and are processed when a computer starts up or a user logs in. They can enforce anything from password complexity to blocking USB drives to deploying software.

⚙️
GPO Processing Order — LSDOU
Later policies override earlier ones (with exceptions)
L

Local GPO

Applied first. Settings configured directly on the local computer. Lowest precedence — overridden by everything else.

S

Site GPO

Applied second. Linked to AD Sites (physical network locations). Rarely used — mainly for network-topology-based settings.

D

Domain GPO

Applied third. Linked to the domain itself. Good for universal settings like password policy and account lockout.

OU

OU GPO (innermost wins)

Applied last — highest precedence. Nested OU GPOs: parent OU first, then child OU. Most specific setting wins. Admins use OU GPOs for department-specific policies.

🔐
Security Settings You Can Enforce via GPO
Minimum password length (recommend 12+), complexity requirements (upper/lower/number/symbol), password history (remember 24), max password age (90 days), account lockout threshold (5 attempts). Applied at Domain level only for domain accounts. Fine-Grained Password Policies (FGPP) allow different policies per group.
Block access to Control Panel, disable USB/removable storage, restrict which applications can run (AppLocker / Software Restriction Policies), force screensaver with password, deploy software packages via MSI, map network drives and printers. Every user setting can be locked down centrally.
Windows Firewall rules, disable NTLM, enforce SMB signing, configure Windows Defender settings, enable audit policies (logon events, object access, privilege use), configure Credential Guard, enable Protected Process Light for LSASS. GPO is your primary security hardening tool.
Run PowerShell/batch scripts at startup, shutdown, logon, or logoff. These scripts live in \\DC\SYSVOL\ — a critical location attackers check for hardcoded credentials. Also used for legitimate tasks: map drives, set printer defaults, clear temp files. Always check SYSVOL during a pentest.
🚩 HTB Checkpoint GPO Abuse — Privilege Escalation

If you have write permissions on a GPO that applies to an OU containing admin accounts or DCs, you can abuse it:

▶Get-GPO -All then BloodHound to find GPO write permissions
▶Use SharpGPOAbuse to add immediate task via GPO → execute as SYSTEM on targets
▶Check \\DC\SYSVOL\domain\Policies\ for GPP files with cpassword (CVE-2014-1812)
▶Get-GPPPassword from PowerSploit — decrypts AES-256 GPP passwords (MS published the key!)
Chapter 08
🗄️ Critical Asset

NTDS.dit — The AD Database

The most sensitive file in your entire Windows environment. Contains every user's password hash, every object, every secret. Here's what's inside and why it matters.

NTDS.dit (NT Directory Services Database) is a Microsoft JET Blue database stored at C:\Windows\NTDS\NTDS.dit on every Domain Controller. It contains every object in Active Directory — users, computers, groups, password hashes, Kerberos keys, and more. This single file, if extracted, gives an attacker the keys to your entire kingdom.

📂
What's Inside NTDS.dit
🗃️

Database Tables

  • datatable — All AD objects (users, computers, groups) and their attributes
  • link_table — Relationships between objects (group memberships)
  • sd_table — Security descriptors (ACLs, permissions)
  • hidden_table — Metadata, schema info
🔑

Password Hashes Stored

  • NT Hash (MD4 of password) — Used for NTLM auth
  • LM Hash (legacy, disabled by default)
  • Kerberos Keys (AES-256, AES-128, DES variants)
  • WDigest (cleartext if enabled — old setting)
  • NTDS.dit is encrypted with the SYSTEM key (Boot Key / SYSKEY)
💉
Extraction Methods
# Method 1: Volume Shadow Copy (local)
vssadmin create shadow /for=C:
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\
       Windows\NTDS\NTDS.dit C:\Temp\
reg save HKLM\SYSTEM C:\Temp\SYSTEM

# Method 2: secretsdump (impacket)
impacket-secretsdump -ntds NTDS.dit -system SYSTEM LOCAL

# Method 3: DCSync (no file needed!)
mimikatz # lsadump::dcsync /domain:corp.local /all

# Crack the hashes
hashcat -m 1000 hashes.txt rockyou.txt
🛡️

Defending NTDS.dit

  • Restrict DC access to Tier 0 admins only
  • Monitor VSS creation events on DCs (Event 8222)
  • Block DCSync: restrict DS-Replication-Get-Changes-All permission
  • Enable Microsoft Defender for Identity — detects DCSync in real-time
